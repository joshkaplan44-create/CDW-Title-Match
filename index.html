What is GCI?

A Golden Contact is a single, trusted, deduplicated view of a person across all CDW systems. The GCI pipeline resolves the same individual appearing in Salesforce, eAccount, event registrations, and email subscriptions into one master record — then enriches it with ZoomInfo data and assigns buyer personas for GTM activation.

The output powers Salesforce, Marketo, MARKO, and Microsoft Fabric — giving every downstream system the same authoritative contact identity.

The Problem We Solved

1
Fragmented identity across 4 systems
The same person existed as separate records in Salesforce, eAccount, Attendee, and Subscriber — with no common key linking them.
2
790K+ unstructured job titles
Free-text title entries like "Sr. Dir IT Ops" and "Senior Director of IT Operations" were treated as different people by segmentation models.
3
No buyer persona signal
Without standardized titles and industry classification, CDW had no systematic way to identify Decision Makers, Champions, or Technical Advisors at scale.

What is EntityID?

EntityID is the primary blocking key used to group candidate record pairs before ML comparison. Rather than comparing every record to every other record (computationally infeasible at 21M scale), Zingg uses EntityID to narrow down which pairs are even candidates. 99.73% coverage means virtually all records participated in deduplication — the 54K without an EntityID were isolated and could not be matched or merged.

What is a Collision?

A collision occurs when two source records are assigned to the same GCI but appear to represent different people — a false positive merge. At 2.88%, the collision rate is well within acceptable thresholds for an ML-based identity resolution system at this scale.

Collisions are flagged for review rather than automatically corrected, preserving auditability. The 97.12% clean rate reflects the precision of the Zingg model's match scoring and the blocking strategy used to constrain candidate pair generation.

What does "Net New" mean?

Net new contacts are GCIs that did not exist in the previous pipeline run — representing newly discovered people introduced through the latest data ingestion cycle. This could be new Salesforce contacts, new event attendees, new email subscribers, or new ZoomInfo records surfaced since the last run. A 10.96% net new rate indicates healthy, ongoing data growth in CDW's contact universe.

How Zingg Works
1
Blocking
Zingg groups records into candidate pairs using blocking rules — dramatically reducing the comparison space from O(n²) to manageable candidate sets.
2
Labeling
A widget surfaces candidate pairs for human review. Reviewers mark each as Match, Non-Match, or Uncertain. GCI trained on 1,200 labeled samples (200 confirmed matches).
3
Training
Zingg trains a probabilistic ML model on the labeled pairs, learning which field combinations are reliable signals for a match.
4
Match
The trained model scores all candidate pairs. Records above the threshold are assigned the same ZINGG_ID, forming a cluster. Each cluster = one golden contact.
5
Survivorship
For each cluster, field-level survivorship logic picks the best value per field using source rank (Salesforce > eAccount > Attendee > Subscriber) and frequency-within-source.

Incremental vs. Full Run

Zingg determines whether to run as a first run or incremental run by checking whether the model output table path already exists in the GCI Lakehouse. If the path doesn't exist, it's treated as a first run and rebuilds all clusters from scratch.

Critical: The model output table path in the utils notebook must be accurate at all times. An incorrect path causes the pipeline to run as a first run, overwriting existing clusters.

The current dataset (full run) produced the bridge table with ~18.4M rows. Future incremental runs will process only new/changed records and merge results into the existing cluster state.

Two-Phase Matching Approach
1
Build Canonical Schema
Combined ZoomInfo's Job Function and Management Level fields into 745 structured titles spanning 148 job functions and 5 seniority levels.
2
Vectorize with TF-IDF
All 790K CDW titles converted to TF-IDF vectors. An abbreviation expansion dictionary (80+ terms) preprocessed "Sr.", "Dir.", "VP", etc. before vectorization.
3
Phase 1 — Cosine Similarity
Each title matched against the canonical schema using cosine similarity. Records scoring ≥60% assigned the best-fit canonical title. 267,816 records (33.9%) matched.
4
Phase 2 — Hierarchy Keyword
Remaining 522,410 unmatched records passed to a keyword classifier targeting management-level tiers (Manager, Director, VP, C-Level, Board). Recovered 58,737 records (7.4%).

Persona Algorithm
4-Stage
Industry keyword → NAICS → semantic
4-Stage Persona Derivation
1
Industry Keyword Match
Match contact's industry against 12 vertical keyword lists. Confidence 0.95 when matched. Determines which solution categories are relevant for this contact's industry.
2
NAICS Exact Match
Use ZoomInfo's NAICSCode to look up industry from the 1,579-entry NAICS reference table. Exact match against industry classification.
3
NAICS Fuzzy Match
If exact NAICS match fails, apply fuzzy matching against NAICS descriptions using TF-IDF similarity. Broader coverage at slightly lower confidence.
4
Semantic Embedding
SentenceTransformer (all-MiniLM-L6-v2) generates semantic embeddings for titles that couldn't be matched via NAICS. Cosine similarity ≥0.5 threshold for persona assignment.

ReferenceRoleMaster Generation

The persona classification system uses a GPT-4o-mini model to generate a ReferenceRoleMaster lookup table offline. For each combination of job title + industry + company size, the model determines:

1. Which solution categories the role cares about (determined first)
2. Which personas apply to each selected category (determined second)
3. Company size gating applied to restrict strategic influence for non-leadership roles

The lookup table is pre-computed offline and used at inference time — avoiding LLM calls at scale during pipeline runs.

How We Built the Identity Resolution Layer
1
Source System Ingestion
Built 4 ingestion pipelines (Salesforce, eAccount, Attendee, Subscriber) using a shared base pipeline pattern. All pipelines share the same code structure, configuration system, and logging framework.
2
Zingg Integration
Zingg is imported as a Python package via zingg.client. Arguments passed programmatically for each phase (label, train, match, incremental). Zingg files deployed as custom packages in the IDR_ENV Fabric environment.
3
ZoomInfo Enrichment Staging
Built a model ingestion pipeline that applies ZoomInfo enrichment logic and creates a staged ZoomInfo table. This converts the raw ZoomInfo match output into the input format expected by Zingg.
4
Post-Processing & Survivorship
After Zingg clustering, survivorship logic applies source rank (Salesforce > eAccount > Attendee > Subscriber) and frequency-within-source to select winning field values for each cluster.
5
Orchestration
All notebooks callable from a single entry point (main_dev_v1) via an orchestrator class. Each phase (run_ingestion, run_model, run_post_processing, run_interactions) is a single function call.
How We Built the Title Matching Layer
1
Abbreviation Expansion
Built a dictionary of 80+ abbreviation expansions (Sr. → Senior, Dir. → Director, VP → Vice President, etc.) applied before vectorization to normalize common shorthand.
2
TF-IDF Vectorization
All 790K CDW titles and 745 canonical ZoomInfo titles vectorized using sklearn TfidfVectorizer. Word-level tokenization with sublinear term frequency scaling.
3
Cosine Similarity Matching
Sparse matrix cosine similarity computed against all 745 canonical titles. Best-match canonical title and score stored for each record. Threshold: 60% for acceptance.
4
Hierarchy Keyword Fallback
Records below the 60% threshold passed to a keyword classifier targeting management-level tier keywords (Manager, Director, VP-Level, C-Level, Board Members). Recovered 7.4% additional coverage.
How We Built the Persona Layer
1
ZoomInfo Canonical Reference (Offline)
Used GPT-4o-mini with a detailed system prompt to generate the ReferenceRoleMaster table offline — mapping each title + industry + company size combination to solution categories and personas.
2
Industry Classification
4-stage vertical derivation: industry keyword match (confidence 0.95) → NAICS exact match → NAICS fuzzy match → semantic embedding fallback using SentenceTransformer all-MiniLM-L6-v2.
3
Title-to-Persona Mapping
Standardized title from Phase 1/2 looked up in ReferenceRoleMaster. Cosine similarity threshold 0.5 for semantic matching. Company size bucket applied as final gating filter.
4
PersonaContactMaster Output
Results written to PersonaContactMaster. One row per persona per contact (contacts can have multiple rows for multiple personas). Joined to Dimension_ContactMaster on ContactMasterID.
Key Design Decisions
Two Lakehouses (by design)
Source data in Model Training Lakehouse (NH), outputs in GCI Lakehouse. Separation ensures no accidental overwrites of source data during pipeline runs. Path to merge into one lakehouse is planned.
Configuration-First Architecture
All paths, table names, and source URLs live in a single utils notebook. Changing lakehouses or table names requires editing one file — not hunting through notebook code.
Schema Notebooks
Each output table has its own schema notebook. Adding a field to a table doesn't break the model — it only reads required fields. New fields must be explicitly added to the relevant schema notebook.
Offline LLM Lookup Table
GPT-4o-mini runs offline to pre-compute the ReferenceRoleMaster table. At inference time, persona assignment uses the lookup table — no LLM calls per record at scale.

Identification Approach

Contact-level identification: For contacts matched to ZoomInfo, job function and management title fields were used alongside an LLM to determine IT decision maker status. For the remaining contacts without ZoomInfo data, raw job titles were analyzed using an LLM to classify decision maker authority. Both populations were filtered to IT and Engineering titles and departments.

Organization-level identification: The same two-population approach was applied with the added dimension of organization name. Decision maker contacts were rolled up to their associated organization, yielding the count of unique accounts with at least one contactable IT decision maker in CDW's database.
